## 一、背景说明

今天对整个小程序的页面架构和核心交互逻辑做了一次系统性复盘，发现现有代码在几个关键环节存在逻辑缺失或定义不清的问题。这次不是推翻重来，已有的页面结构、UI 规范、组件规范全部保留，需要调整的是若干页面的**职责定义、跳转逻辑和交互细节**。

请在动手前完整阅读本说明，理解每一处修改的原因，再对照 `prd032402.md` 确认细节。

## 二、这次调整涉及的核心变化

### 变化 1：`pages/mine/index` 的定位重新明确

**原来的问题：**
`pages/mine/index` 和 `pages/profile/index` 两个页面的内容高度重叠，导致同一套 UI 要维护两份代码，容易出现不一致。

**调整后的设计：**
- `pages/profile/index` 是**唯一的用户主页页面**，通过 URL 参数 `uid` 判断当前访问者是本人还是他人，从而控制显示内容的范围：
  - 本人视角：显示完整内容，包括「编辑资料」按钮、快捷入口（我的创建/我的申请/我的足迹/我的心愿）、右上角设置图标
  - 他人视角：只显示头像、昵称、UID、性别、身份 tag、签名、关注/粉丝/获赞占位、档案馆模块，隐藏所有私有操作入口
- `pages/mine/index` 作为 **Tab2 的壳页面**保留（因为微信 TabBar 页面不支持传参，无法直接用 switchTab 跳到带 uid 参数的 profile 页）。它的唯一职责是：在 `onShow` 里读取当前登录用户的 uid，然后渲染与 `profile/index` 本人视角完全相同的内容。两个页面**共用同一套组件逻辑**，不各自维护一套 UI。

**需要做的事：**
- 重构 `pages/mine/index`，使其内容与 `pages/profile/index` 本人视角保持一致，抽取公共部分为组件复用
- `pages/profile/index` 增加 `uid` 参数判断逻辑，区分本人/他人视角的显示内容

### 变化 2：`pages/home/detail`（招募详情页）的底部按钮逻辑重写

**原来的问题：**
底部按钮只考虑了申请人的几种状态，没有处理「发布人本人看自己招募」和「招募已发车」这两种情况。

**调整后的完整状态表：**

| 访问者身份       | 招募状态       | 按钮文字     | 样式         | 点击行为                     |
| ----------- | ---------- | -------- | ---------- | ------------------------ |
| 游客          | recruiting | 申请加入     | 深色宽按钮      | 触发授权弹窗                   |
| 已登录他人 · 未申请 | recruiting | 申请加入     | 深色宽按钮      | 先做状态检查，再提交申请             |
| 已登录他人 · 待审核 | recruiting | 等候审核中    | 灰色，不可点     | 无响应                      |
| 已登录他人 · 已通过 | recruiting | 已通过      | 绿色宽按钮      | 弹出联系方式弹窗                 |
| 已登录他人 · 已拒绝 | recruiting | 未通过      | 灰色，不可点     | 无响应                      |
| **发布人本人**   | recruiting | **申请加入** | **灰色，不可点** | **弹提示「你是本次招募的发布人，无法申请」** |
| 任何人         | finished   | 已发车      | 灰色，不可点     | 无响应                      |

关键点：发布人本人进入详情页，按钮显示为**灰色不可点的「申请加入」**，视觉上暗示此操作对他不适用，点击后弹提示说明原因。不要隐藏按钮，也不要换成其他文字，保持布局统一。

**同时新增：发布人视角下的「管理招募」入口**
- 位置：页面内容区（不是导航栏），右上角分享图标附近，用文字链接形式呈现
- 仅当当前登录用户 uid === 该招募的 creatorId 时显示
- 点击跳转 `pages/mine/detail?id=xxx`

**新增：申请时的状态检查（乐观锁）**
用户点击「申请加入」提交申请的瞬间，先查一次该模组的当前 status：
- 仍为 `recruiting`：正常提交申请，流程不变
- 已变为 `finished`：弹提示「该招募已发车，无法申请」，底部按钮同步变为灰色「已发车」，不提交申请

### 变化 3：`pages/publish/form`（发布/编辑表单）的路径和逻辑调整

**路径变更：**
发布表单的路径从 `pages/publish/index` 改为 `pages/publish/form`，同时它**不再是 TabBar 页面**，改为普通子页面（用 `uni.navigateTo` 跳转，有原生返回箭头）。请同步更新 `pages.json`，将其从 TabBar 配置中移除。

**编辑模式全字段回填：**
当携带 `id` 参数进入时（编辑模式），表单所有字段必须回填，**包括联系方式字段**。原来联系方式不回填的设计只适用于新建模式，编辑模式下如果联系方式为空，用户可能忘记填写，导致申请人通过审核后看不到联系方式，这是一个数据丢失 bug。

**新增：返回键弹窗逻辑（仅新建模式）**
拦截原生返回键（`onBackPress`），根据表单填写情况分三种处理：

- **情况一：模组名称已填写** → 弹确认窗，内容「是否保存为草稿？」，两个按钮：「保存草稿」（等效底部保存草稿按钮，跳转「我的创建」）/ 「不保存」（直接返回，不生成卡片）
- **情况二：模组名称未填，但其他字段有内容** → 弹提示窗，内容「至少填写模组名称才能保存草稿」，两个按钮：「去填写」（留在表单，光标聚焦到模组名称字段）/ 「放弃」（直接返回，不保存）
- **情况三：表单完全为空** → 直接返回，不弹窗

编辑模式下，因为模组名称必然已有内容，返回键触发时直接走情况一的逻辑。

**新增：「发布」按钮确认提示**
点击底部「发布」按钮后，在跳转前弹一个轻提示（Toast 或简单弹窗），文案「发出去喽」，让用户明确感知已发布成功，而不是静默跳转。

### 变化 4：发车逻辑的联动处理

**新增：发车时自动清算 pending 申请**
发布人点击「确认发车」后，系统需要自动将该招募下所有状态仍为 `pending` 的申请记录，统一改为 `rejected`。这个操作在发车确认的同一个方法里处理，发布人不需要手动逐一拒绝未处理的申请。

发车确认弹窗的文案也需要同步更新，改为：
> 「当前已通过 X 人，发车后招募将关闭，未处理的申请将自动标记为未通过，确认发车？」

### 变化 5：`pages/mine/created`（我的申请）卡片增加发车后状态

**原来的问题：**
「我的申请」列表只有三种状态标签（审核中 / 已通过 / 未通过），没有考虑招募已发车的情况。

**调整后：**
发车后，系统将所有 pending 自动归并为 rejected（见变化 4），因此发车后申请人只会看到两种状态组合：

| 发车后显示 | 文字颜色 | 背景色 | 含义 |
|---|---|---|---|
| ✓ 已通过 · 已发车 | #2a7a4a | #e8f5ee | 上车成功，团已出发 |
| 未通过 · 已发车 | #666666 | #e8e8e8 | 未通过或未被处理，团已走 |

实现思路：判断该申请对应的模组 status 是否为 `finished`，若是则在原有审核状态标签后追加「· 已发车」文字。

---

### 变化 6：三个页面新增 `onShow` 数据刷新

以下三个页面需要在 `onShow` 生命周期钩子中重新读取一次数据：

| 页面 | 原因 |
|---|---|
| `pages/home/index` | 发布人在别处发车后，大厅卡片需要在用户切回来时消失 |
| `pages/home/detail` | 招募状态可能在用户停留期间发生变化，底部按钮需同步 |
| `pages/mine/detail` | 从审核页返回后，已通过/待审核人数需要立即更新 |

## 四、建议的修改顺序

1. **先改 `pages.json`**：将发布表单从 TabBar 移除，路径改为 `pages/publish/form`，确认 TabBar 只剩 2 个入口
2. **改 `pages/mine/index`**：重构为壳页面，与 profile 共用组件
3. **改 `pages/home/detail`**：底部按钮完整状态表 + 发布人「管理招募」入口 + 乐观锁检查 + onShow 刷新
4. **改 `pages/publish/form`**：编辑模式回填联系方式 + 返回键弹窗逻辑 + 发布确认提示
5. **改发车逻辑**（`pages/mine/detail`）：发车时自动清算 pending + 弹窗文案更新 + onShow 刷新
6. **改 `pages/mine/created`**：我的申请列表发车后状态标签
7. **改 `pages/home/index`**：onShow 刷新

每一步改完后最好在模拟器里跑一下对应的用户流程，再进行下一步。

## 五、完整的页面跳转关系（供参考）

```
招募大厅（Tab1）
  └─ 点卡片 → home/detail（申请人视角）
       ├─ 点发布人头像 → profile/index?uid=xxx（他人视角）
       ├─ 点「申请加入」→ 提交申请（含状态检查）
       ├─ 点「已通过」→ 弹联系方式弹窗
       └─ 点「管理招募」（仅发布人本人可见）→ mine/detail?id=xxx

我的（Tab2）= mine/index（壳）= profile/index 本人视角
  ├─ 点「编辑资料」→ profile/edit
  ├─ 点「我的创建」→ mine/created?tab=created
  │    └─ 点卡片 → mine/detail?id=xxx（发布人详情）
  │         ├─ 点「编辑」→ publish/form?id=xxx（编辑模式，全字段回填）
  │         ├─ 点「审核」→ mine/review
  │         │    └─ 点申请人头像 → profile/index?uid=xxx（他人视角）
  │         └─ 点「发车」→ 弹确认窗 → 确认 → 自动清算 pending → 跳 mine/created
  ├─ 点「我的申请」→ mine/created?tab=applied
  │    └─ 点卡片 → home/detail?id=xxx（申请人视角）
  └─ 档案馆·演绎记录 → profile/log_detail

悬浮「+」按钮（全局）
  └─ 点击 → 登录检查 → publish/form（新建模式，无 id 参数）
```
