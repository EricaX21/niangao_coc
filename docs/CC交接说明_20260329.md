## 第一批：修正现有逻辑（优先级最高）

### 1. P3 发布/保存后改用 redirectTo

**影响文件：** `pages/publish/form.vue`

**现在的问题：** P3 发布或保存草稿后使用 `navigateTo` 跳转到 P8，导致用户在 P8 点返回时又回到一个空的发布表单。

**改为：**
- 点击「发布」→ 弹「发出去喽」确认 → 确认后 `uni.redirectTo({ url: '/pages/mine/created?tab=created' })`
- 点击「保存草稿」→ `uni.redirectTo({ url: '/pages/mine/created?tab=created' })`
- 返回键弹窗中选「保存草稿」→ 同样使用 `uni.redirectTo`

`redirectTo` 会替换页面栈顶部的页面，不增加栈深度。效果是用户在 P8 点返回时直接回到 P3 之前的页面（比如 P1 或 P9），而不是回到空表单。

---

### 2. P9 草稿状态「发布」按钮加确认弹窗

**影响文件：** `pages/mine/detail.vue`

**现在的问题：** P9 在 draft（草稿）状态下点击「发布」按钮直接将状态变为 recruiting，没有确认提示。

**改为：** 点击「发布」后弹出确认提示「发出去喽」，用户确认后再将状态变为 recruiting（招募中）。行为与 P3 表单的「发布」按钮保持一致。

---

### 3. P2 底部按钮：已发车后已通过用户仍可查看联系方式

**影响文件：** `pages/home/detail.vue`

**现在的问题：** P2 底部按钮在 finished（已发车）状态下统一显示灰色「已发车」不可点。但已通过审核的申请人在发车后应该仍能查看联系方式。

**改为：** 在底部按钮的状态判断逻辑中，增加一个分支：

```
if (招募状态 === 'finished' && 当前用户申请状态 === 'approved') {
  // 显示绿色按钮「✓ 已通过 · 已发车」，可点击，弹联系方式弹窗
} else if (招募状态 === 'finished') {
  // 显示灰色「已发车」，不可点
}
```

具体状态对照（仅新增行）：

| 招募状态 | 访问者 | 申请状态 | 按钮文字 | 可点击 | 点击行为 |
|---|---|---|---|---|---|
| finished | 已登录他人 | approved | ✓ 已通过 · 已发车 | ✅ | 弹联系方式弹窗 |
| finished | 其他所有人 | — | 已发车 | ❌ | — |

### 4. P9 发车成功后改用 redirectTo

**影响文件：** `pages/mine/detail.vue`

**现在的问题：** P9 发车成功后跳转 P8，如果使用 navigateTo，用户在 P8 点返回会回到一个已经发车的 P9 页面（虽然 onShow 会刷新，但栈里多了一层无意义的页面）。

**改为：** 发车确认后使用 `uni.redirectTo({ url: '/pages/mine/created?tab=created' })`。

### 5. 退出登录功能

**影响文件：** `pages/profile/index.vue`（或 `pages/mine/index.vue` 中的对应组件）、`src/stores/` 相关 store

**现在的状态：** 右上角设置图标点击后 Toast「即将开放」。

**改为：**
1. 点击设置图标 → 弹确认窗「确定退出登录？」
2. 用户确认 → 清空 store 中的用户数据（userStore 重置为未登录态）
3. 清空后 → `uni.switchTab({ url: '/pages/home/index' })`，回到招募大厅游客态

## 第二批：核心新增功能

### 6. 用户主页改为四层吸顶布局

**影响文件：** `pages/profile/index.vue`、`pages/mine/index.vue`

**这是本次改动量最大的一项。** 用户主页（P4 / P7）的页面结构需要改为以下四层：

| 层级 | 内容 | 实现方式 |
|---|---|---|
| 第一层 | 个人信息区（头像、昵称、UID、性别、身份 tag、签名、关注/粉丝/获赞） | 正常文档流，跟随页面滚动 |
| 第二层 | 快捷入口四格（我的创建/申请/足迹/心愿），仅本人视角 | 正常文档流，跟随页面滚动 |
| 第三层 | 档案馆 tab 栏（演绎记录 / 属性面板 / 人物卡 / 跑团饭） | **`position: sticky; top: 0; z-index: 10;`** |
| 第四层 | 当前 tab 的内容区 | 正常文档流 |

**交互效果：** 用户往下滑动时，第一层和第二层自然滚出屏幕顶部。第三层 tab 栏到达顶部后"粘住"不动。第四层内容在 tab 栏下方继续滚动。效果类似小红书个人主页。

**档案馆 tab 切换：** 切换 tab 时替换第四层内容区。MVP 阶段只有「演绎记录」tab 有真实内容，其他三个 tab（属性面板 / 人物卡 / 跑团饭）显示占位文案「即将开放」。

**注意：** 之前的档案馆是四个模块卡片的形式，现在改为 tab 栏 + 内容区的形式。这是结构性变更。

### 7. 演绎记录缩略卡片在主页内展示

**影响文件：** `pages/profile/index.vue` 档案馆区域

**依赖：** 第 6 项完成后才能做。

在档案馆的「演绎记录」tab 内容区中：
- 展示该用户所有演绎记录的缩略卡片，**不做截断，全部展示**
- 每张缩略卡片包含：标题 + 模组名称 tag + 其他 tag + 正文前两行预览文字 + 创建日期
- 点击某张缩略卡片 → `uni.navigateTo` 跳转 `pages/profile/log_detail?id=xxx`
- **本人视角**：缩略卡片列表底部显示「+ 新增」按钮，点击跳转 `pages/profile/log_detail`（无 id 参数，新建模式）

- 数据来源：从 mockData 中读取当前 uid 对应的所有 Log 记录

### 11. 验证 P1 筛选状态保留

P1 是 TabBar 页面，switchTab 切换回来时页面不会被销毁，筛选状态理论上天然保留。请验证以下场景：

1. 在 P1 选择筛选条件（如"COC"）→ 点击卡片进入 P2 → 返回 P1 → **筛选条件应仍为"COC"**
2. 在 P1 选择筛选条件 → 切换到 Tab2 → 切换回 Tab1 → **筛选条件应仍保留**

如果已经是对的，不需要改代码。如果不对，需要确保筛选状态存储在页面的响应式数据中而非临时变量中。
