## 二、本阶段重要决策档案

> 完整论证见 `prd_v1_5.md` 第 20 章。这里记「一句话版」，新对话读这些就够衔接。

| 决策 | 内容 | 为什么 |
|---|---|---|
| **插件，不是平台** | 产品动作是"帮 KP 把东西做好拿去群里用"，不是"大家都来我这" | 合伙人判断：同类项目死于"想商业化→弄复杂"；三板块中发布+美化本是一体 |
| **首页即编辑器** | 启动页从大厅换成发车表单，TabBar 两 tab 无中间按钮 | 先群后厅：冷启动期首页的实际访客几乎只有想开团的 KP |
| **大厅降级不删除** | 从首页「逛大厅」按钮进入，去掉搜索和筛选 | "要有但不显眼"；搜索+筛选+卡片流是电商骨架 |
| **必填只有三项** | 名称 / PL 人数 / 联系方式；手填以点选为主 | 北极星"发布<10秒"取决于控件类型，不只是字段数量 |
| **弹性时间** | 日期和时段独立、留空即可协商、全空才显示"时间待商量" | 真实招募中"时段定日期议"（闇暗山）和"都待定"（留言板团）大量存在 |
| **空着可以，乱填不行** | 粘贴识别的底线原则 | 解析失败必须是"没帮上忙"，不能是"帮了倒忙" |
| **粘贴识别用规则不用大模型** | 纯前端正则，样本积攒在手册里，规则不够用了再上 LLM | 零成本零延迟；规则失败案例本身就是将来上模型的依据 |
| **模组库（爬站自动填充）暂缓** | 不做 | 版权是前置问题；改编团永远不在库里 |
| **明确不加的容器** | 车卡村规/推荐技能/平台设备/时代背景不进表单 | 卡片是钩子不是说明书，细节留在群公告 |

## 三、接下来的路线图（按顺序）

> 时间是粗估，按"你的可投入时间"伸缩。原则：**每个阶段有明确的"完成标志"，没达到就不进下一阶段。**

### 阶段 1：验收与修补（现在，1~2 次会话）

把上面的验收清单逐项过完，发现的问题当场发给 AI 修。
**完成标志：清单全勾，模拟器里核心闭环无报错。**

### 阶段 2：卡片/海报设计（1~2 周，节奏取决于设计沟通）

和设计合作方定卡片视觉方向（参考"电影票根"意向），产出设计稿或参考图 → AI 实现 Canvas 绘制 → 真机看效果调整。
**注意：这是 V1.5 的门面**——发车直出的那张卡片是用户甩进 QQ 群的东西，它的质感直接决定"美观传播"这个板块成不成立。
**完成标志：你和设计方都认可的卡片，在真机上生成、保存、发群都清晰好看。**

### 阶段 3：双人真机联调（执行半天，修补几天）

先让 AI 按 V1.5 新流程修订 `V1.4联调测试清单.md`，再和队友（体验用户）在真机上跑完整闭环：发布 → 群里点卡片 → 申请 → 审核 → 发车 → 查看联系方式。
**完成标志：两个真实账号在真机上跑通闭环，P0 级问题清零。**

### 阶段 4：小范围真实用户试用（2~4 周）

拉几个真实 KP 用起来。重点收集两样东西：**真实招募文本样本**（喂给粘贴识别手册）和**"哪里觉得像平台/复杂了"的反馈**（验证合伙人的判断）。
**完成标志：≥5 个非自己人的 KP 发过车；样本库和反馈清单有实质积累。**

### 阶段 5：上架准备（与阶段 2-4 并行推进）

ICP 备案、小程序类目、隐私协议、用户协议——行政事项，不占开发时间，但周期长，**建议现在就启动备案**。

## 四、常规开发节奏（给非开发者的参考）

### 4.1 开发的基本循环

每一个改动，无论大小，都走同一个循环：

```
说清楚要什么 → 讨论方案（可能来回几轮）→ 实现 → 你验收 → 修补 → 提交保存
```

**核心经验：每轮改动越小越好。** 改动越大，出错面越大，验收越难，出了问题越难定位。这次 V1.5 是特例（方向性改版必须整体动），日常迭代应该是"一次会话解决一到两个明确的问题"。

### 4.2 先做什么后做什么的判断规则

1. **先跑通，再好看**——功能闭环 > 视觉打磨（这就是为什么海报占位、设计放后面）
2. **先核心链路，后旁枝**——发布→申请→发车这条命脉优先，档案馆足迹等靠后
3. **先修"堵路"的，再修"难看"的**——按钮被盖住是堵路（P0 级），选中态颜色不对是难看（P1 级）
4. **动结构之前先动纸面**——大改先在 PRD/讨论里想清楚再写代码（这次我们先出方向备忘再动手，返工少）

### 4.3 各类工作的大致耗时感受（AI 协作下）

| 工作类型 | 耗时感受 | 例子 |
|---|---|---|
| 文案/文档修改 | 几分钟 | 改按钮文字、更新 PRD 某节 |
| 单点 UI 调整 | 会话内当场完成 | 吸顶行高度、按钮排布 |
| 修 bug | 不确定性最大：几分钟~几轮 | 定位难的（如选中态陈年 bug）靠"给出复现步骤+截图"提速 |
| 新功能（单页面） | 1 次会话 | 演绎记录删除按钮+云函数 |
| 结构性改版 | 1 个长会话 + 后续几轮修补 | 这次的首页改版 |
| 设计类工作 | 取决于你给方向的速度 | 卡片排版：AI 能实现，但"要什么感觉"只有你能定 |

### 4.4 改动前值得问的四个问题

1. **这减少了什么复杂度？**（合伙人的警告：加东西容易，减东西难）
2. **用户的数据会不会丢？**（涉及表单/草稿/存储的改动必问）
3. **老链路会不会被改断？**（每次结构改动后要回归测核心闭环）
4. **要不要动云函数？**（动了就要重新部署，多一步操作）

## 五、和 AI 高效协作的方法

### 5.1 什么样的输入最有效

- **截图 + 复现步骤**：「我点了X，出现了Y，我预期是Z」——这次选中态 bug 和按钮被盖住都是这样秒定位的
- **真实样本**：三段真实招募文本直接把解析器质量顶上去了，比任何描述都有用
- **参考物**：「像醒图那样」「像淘宝地址粘贴那样」——比抽象描述效率高得多
- **说出你的犹豫**：「我们还在商量，没定」是有效信息，AI 会帮你分析利弊而不是直接实现

### 5.2 你不用懂但要知道的固定流程

- **每次构建后云函数目录会被清空**，需要重拷（跟 AI 说一声就行）
- **新增云函数必须在开发者工具里手动部署**（右键 → 创建并部署：云端安装依赖）
- **改完的代码要提交推送 GitHub**（AI 会做，但值得确认它做了——没提交的工作是悬空的）
- **改解析规则前要跑回归测试**（`npm run test:parse`，AI 会自动做）

### 5.3 开新对话的衔接话术（可直接复制）

> 请先读项目根目录的《开发指南与进度参考.md》了解现状和决策背景，再读 CLAUDE.md 的开发规范。需求以 prd_v1_5.md 为准，粘贴识别规则以《粘贴识别规则手册.md》为准。今天我想做：______

### 5.4 哪些事 AI 会自己定，哪些会来问你

**AI 自己定**：技术实现方式、代码结构、bug 修法、文档措辞
**会来问你**：产品方向选择（如"吸顶还是悬浮"）、要花钱的事（API 密钥）、删除/覆盖已有内容、对外发布类操作
**只有你能做**：微信扫码登录、开发者工具里的部署点击、和真人（合伙人/设计/队友）的沟通
