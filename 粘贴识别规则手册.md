# 粘贴识别规则手册

> 本手册与 `scripts/test-parse.mjs` 互为镜像：**手册管「为什么」，测试管「是否还对」**。
> 每次调整规则或新增样本，两边同步更新，然后跑 `npm run test:parse`。
> 解析器实现：`src/utils/parseRecruitText.js`；时间展示：`src/utils/formatTime.js`。

## 总原则

**空着可以，乱填不行。** 识别不确定的信息宁可留空——留空在弹性时间模型下就是「可协商」，是有效状态而非缺失。解析器的失败模式必须是"没帮上忙"，绝不能是"帮了倒忙"。

---

## 一、弹性时间模型（展示逻辑）

日期与时段是**两个独立维度**，各自有「确定 / 可协商」两种状态：

- 闇暗山（S3）：时段确定（晚7-10）+ 日期可协商 → 时段是重点，日期不是
- 「仅限双休」类：日期确定 + 时段可议
- 寂静之声（S2）：两维都可协商（交卡就开、留言板）

**实现上不加任何字段**：留空即可协商，`formatGameTime` 按四态渲染。「待商量」只在两维全空时兜底出现——至少一维有值时只展示有值的那维，不提「待商量」（产品方定：避免文案过于刻意）。

| gameDays | startTime | 卡片显示 | 依据 |
|---|---|---|---|
| 有 | 有 | `周六、周日 20:00-23:00` | 常规团 |
| 有 | 空 | `周六、周日` | 日期定、时段可议 |
| 空 | 有 | `19:00-22:00` | S3：时段定、日期可议 |
| 空 | 空 | `时间待商量` | S2：留言板团 |

附加规则：
- **去掉「每」前缀**：原「每周日 20:00」对 S1 这类 7月19日 的单次团是错误信息；「周日 20:00」对单次团和周更团都中性成立。
- **7 天全选渲染为「每天」**（S6）。
- 排序时周日排最后：「周六、周日」比「周日、周六」符合表达习惯。

**发车必填仅三项**：模组名称 / PL 人数 / 联系方式。时间两维度填了就展示、没填就是可协商。

## 二、样本库

新增样本流程见文末。当前样本在 `scripts/test-parse.mjs` 中有完整原文与预期断言，此处记录每段的**关键识别点与决策**。

### S1 · 深水城：小猫救援（5r 语音单次团）

| 识别点 | 结果 | 说明 |
|---|---|---|
| `《深水城：小猫救援》` | 模组名称 | 书名号优先 |
| `5r` | 规则 → DND5e | 5r 是圈内对 DND 五版的常用简称 |
| `语音短团` | 方式 → 语音 | |
| `7月19日（周日）` | 跑团日 → 周日 | 括号里的周X；裸日期「7月19日」不换算星期（需要年份假设，不做） |
| `北京时间20:00` | 开始 20:00 | HH:mm 明确格式 |
| `人数：4-5` | PL → 5 | 范围取上限，夹紧 1-6 |
| `DM：大蝠 439510286` | QQ 439510286 | DM/KP 标签后 12 字符内的 5-12 位数字 |
| `模组概要：…` | 简介 | |
| 开卡/平台/设备要求 | **不落任何字段** | 见「明确不做」 |

### S2 · 寂静之声（coc7 留言板团）

| 识别点 | 结果 | 说明 |
|---|---|---|
| `模组:寂静之声` | 模组名称 | 无书名号时认「模组:」标签行 |
| `coc7➕心判` | 规则 → COC7th | 「心判」等村规修饰不影响体系判定 |
| `留言板` | 方式 → 文字；**日期不动** | 留言板团哪几天没定 ≠ 每天；留空即可协商 |
| `交卡就开，固定时间14-5任选` | **时段留空** | 「14-5」无晚/下午前缀不做 12 小时转换；本来就是任选 |
| `推荐游戏人数：1-2 人` | PL → 2 | |
| `4-10h` | 时长 | |
| `1970 年` | **不落时长** | 年份数字防误读（N1 反例） |
| 无任何号码 | 联系方式留空 | 不许编造 |

### S3 · 闇暗山（时段确定、日期待商量——弹性时间模型的触发样本）

| 识别点 | 结果 | 说明 |
|---|---|---|
| `模组:《闇暗山》` | 模组名称 | 标签行内的书名号 |
| `coc7，天命五选一，购点480不含运` | 规则 → COC7th；**480 不落任何字段** | 「购点480」是车卡村规（属性点购买制），不是时间 |
| `技能:80/60` | 不落任何字段 | 同上，村规数字 |
| `kp:苏百里（1736653936）` | QQ 1736653936 | 括号内纯数字兜底规则 |
| `留言板，每天不低于三条` | 方式 → 文字；**日期不动** | **「每天」在此指发言频率不是跑团日**——只有「每天+紧跟时间」（如「每天晚8点」）才视为跑团日全选 |
| `也可以晚7-10集中时段` | 19:00-22:00 | 带「晚」前缀的口语区间才做 12 小时转换 |
| `预计总时长12小时左右` | 时长 12小时 | |
| `前情提要：…` | 简介 | 前情提要/简介/概要 三个标签都认 |

### S4 · 一句话短招募（合成样本）

`COC7《迷雾中的呼唤》周六晚8点 语音 3PL 有意私聊 QQ 123456789`
→ 全字段命中：名称/COC7th/语音/周六/20:00/3PL/QQ。「晚8点」走口语单点规则。

### S5 · 双休 + 微信号（合成样本）

`《海底疑云》双休晚8点半开团 4PL 微信: kp_hello2024`
→ 「双休/周末」= 周六+周日（产品方提的强约束案例）；「晚8点半」→ 20:30；「微信:」后字母开头 6-20 位 → 类型 wx。

### S6 · 真·每天团（合成样本）

`《样本镇》每天晚9点 文字团 2PL 群：987654321`
→ 「每天晚9点」满足「每天+紧跟时间」→ 跑团日全选（渲染为「每天」）；「群：」→ 类型 qqgroup。

### N1 · 反例（守底线）

`开卡规则：购点480不含运，4d6kh3，技能80/60，1970 年背景`
→ 必须**零输出**：无时间、无人数、无时长。车卡村规是有意义的内容（KP 自设规则），但不属于任何表单字段；这些数字一旦误读就是"帮倒忙"。

## 三、规则清单

| # | 匹配模式 | 落入字段 | 依据样本 | 已知边界 |
|---|---|---|---|---|
| R1 | `《…》` 书名号 | 名称 | S1 S3 | 取第一个；截 20 字 |
| R2 | `模组[:：]…` 标签行 | 名称 | S2 | 书名号优先于标签行 |
| R3 | `coc7/七版`、`coc6/六版` | 规则 | S2 S3 | 村规修饰（心判等）不影响判定 |
| R4 | `5r/dnd/D&D` | 规则 DND5e | S1 | 仅映射已知 token；未知体系不乱落 |
| R5 | `语音` | 方式 | S1 | |
| R6 | `文字/留言板/pbp` | 方式 文字 | S2 S3 | |
| R7 | `周X/星期X` | 跑团日 | S1 S4 | 去重；裸日期不换算星期 |
| R8 | `双休/周末` | 跑团日 [六,日] | S5 | |
| R9 | `每天` + 紧跟时间 | 跑团日全选 | S6 | **「每天不低于三条」类频率表述不触发**（S3 雷区） |
| R10 | `HH:mm` | 开始/结束 | S1 | 时 0-23 分 0-59，防日期与 QQ 号误读 |
| R11 | `晚/下午/上午 + X[-~至]Y` | 时段区间 | S3 | **必须带前缀才 +12**；防购点480/4d6kh3/80/60（N1） |
| R12 | `晚/下午/上午 + X点(半)` | 开始时间 | S4 S5 | 半 → :30 |
| R13 | `人数：X(-Y)`、`XPL` | PL 人数 | S1 S2 S3 S4 | 范围取上限；夹紧 1-6 |
| R14 | `X(-Y)小时/h` | 时长 | S2 S3 | 「X天」不识别——「模组内时长：1~2天」是游戏内时间（S2 雷区） |
| R15 | `简介/概要/前情提要[:：]…` | 简介 | S1 S2 S3 | 截到下一个标签行；限 300 字 |
| R16 | `群(号)[:：]数字` | 联系 qqgroup | S6 | 优先级最高 |
| R17 | `vx/wx/微信[:：]…` | 联系 wx | S5 | 字母开头 6-20 位（微信号规范） |
| R18 | `qq[:：]数字`、`DM/KP 附近数字`、`（数字）` | 联系 qq | S1 S3 S4 | 5-12 位；括号兜底最后 |
| R19 | `招/缺/求 KP` | 招募KP 开 | — | 明确出现才置真 |

## 四、明确不做（防复杂化，防遗忘重议）

以下信息真实存在于招募文本中，但**不给表单加容器**——卡片是钩子不是说明书，这些细节留在 KP 的群公告里：

| 类别 | 样本例 | 不做的理由 |
|---|---|---|
| 车卡村规 | 购点480、4d6kh3、天命五选一、技能80/60、3级开卡 | KP 自设规则，高度自由文本，无法结构化；入团后才需要 |
| 推荐技能/职业 | 御三家、侦探记者 | 同上 |
| 平台与设备 | discord+fvtt、需要梯子 | 同上 |
| 时代背景 | 1970美国、现代日本 | 属简介内容，KP 想写会写进简介 |
| 续团安排 | 一天跑不完下周日继续 | 自由文本 |

另：**模组库自动填充（B 方案）暂缓**——版权前置问题未解决前不做，见方向备忘。

## 五、如何添加新样本

1. 把招募原文贴进 `scripts/test-parse.mjs` 的 `cases`，起编号（S7、S8…）。
2. **人工写预期落位**：每个字段应该是什么、哪些留空、为什么（重要：预期是"产品上正确"，不是"当前实现输出什么"）。
3. 跑 `npm run test:parse`。
4. 不符 → 在 `src/utils/parseRecruitText.js` 补规则 → 回归全部样本仍绿 → 在本手册「样本库」和「规则清单」补记。
5. 若某类文本规则实在覆盖不了（识别率低且规则会伤及其他样本），记入下方候选池，作为将来上 LLM 兜底的依据。
